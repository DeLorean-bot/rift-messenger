# 01 — Полная архитектура клиента

Клиент Discord существует в четырёх формах: **desktop** (Electron), **web** (браузер),
**mobile** (React Native), **консольные/встраиваемые**. Все они делят одно **ядро логики**
и различаются только слоем платформы. Ниже — полная архитектура с упором на desktop/web,
так как это самый сложный случай.

---

## 1.1. Слоистая архитектура клиента

```
┌─────────────────────────────────────────────────────────────┐
│  ПЛАТФОРМА (Electron main / RN native / browser)             │
│  окна, IPC, автообновление, оверлей, доступ к ОС             │
├─────────────────────────────────────────────────────────────┤
│  ТРАНСПОРТ                                                    │
│  REST-клиент (HTTP) · Gateway-клиент (WebSocket) ·           │
│  Voice-клиент (WebRTC/UDP) · RPC-сервер (локальный WS)       │
├─────────────────────────────────────────────────────────────┤
│  ДОМЕННОЕ ЯДРО (stores)                                       │
│  Flux-хранилища: Users, Guilds, Channels, Messages,          │
│  Presence, Voice, ReadState, Permissions, Settings           │
├─────────────────────────────────────────────────────────────┤
│  ПРЕЗЕНТАЦИЯ                                                  │
│  React-компоненты, роутер, виртуализация списков,            │
│  тема/дизайн-токены, локализация                             │
└─────────────────────────────────────────────────────────────┘
```

### Что делает
Разделение на слои изолирует бизнес-логику от платформы: одно и то же ядро работает
в браузере и в Electron. Презентация «глупая» — она только читает из хранилищ и
диспатчит действия.

### Как работает
Данные текут **однонаправленно** (Flux): транспорт получает событие → создаётся
**action** → **Dispatcher** передаёт его всем подписанным **stores** → stores обновляют
состояние → React перерисовывает подписанные компоненты. Пользовательский ввод создаёт
обратный action (например, «отправить сообщение»), который идёт в транспорт.

### Технологии
- **React** — представление.
- **Flux** (собственная реализация, не Redux) — Discord использует классический Flux с
  единым Dispatcher и множеством stores.
- **TypeScript**.

### Альтернативы и почему так
- **Redux** — единое дерево состояния; Discord выбрал множество независимых stores, потому
  что данных огромный объём (миллионы сообщений/пользователей) и единый immutable-стор с
  полными перерисовками не масштабируется. Много мелких stores → точечные подписки.
- **MobX/Signals** — реактивность «из коробки», но Flux даёт предсказуемый однонаправленный
  поток и явные actions, что проще отлаживать в приложении такого размера.

## 1.2. Flux: Dispatcher, Stores, Actions

### Что делает
Центральная шина синхронизации состояния клиента.

### Как работает / ожидаемое поведение
1. **Action** — простой объект `{ type: 'MESSAGE_CREATE', message, channelId, ... }`.
2. **Dispatcher** — синглтон. `dispatch(action)` синхронно вызывает всех
   зарегистрированных обработчиков. Поддерживает `waitFor(store)` — упорядочивание
   зависимостей между stores (например, MessageStore ждёт ChannelStore).
3. **Store** — держит срез состояния + `emitChange()`. Компоненты подписаны через
   хук/HOC (`useStateFromStores`), который перерисовывает только при изменении нужного среза.

### Что происходит на клиенте
- Gateway-событие `MESSAGE_CREATE` → action того же типа → `MessageStore` добавляет
  сообщение в массив канала, `ReadStateStore` инкрементирует счётчик непрочитанного,
  `ChannelStore` поднимает канал в списке → перерисовываются только затронутые компоненты.
- Оптимистичные действия: при отправке сообщения клиент сразу создаёт локальное сообщение
  со статусом «sending» и `nonce`, диспатчит `MESSAGE_CREATE` локально; когда с сервера
  приходит настоящее — сопоставляет по `nonce` и заменяет.

### Технологии
Собственный `Dispatcher` (~аналог Facebook Flux Dispatcher), stores как ES-классы.

### Альтернативы
- Прямое обновление React-состояния через контекст — не тянет объём и приводит к каскадным
  перерисовкам. Flux + точечные подписки решает именно это.

## 1.3. Ключевые хранилища (stores)

| Store | Хранит | Заполняется из |
|-------|--------|----------------|
| `UserStore` | пользователи (id → объект) | READY, GUILD_MEMBER_ADD, MESSAGE_CREATE |
| `GuildStore` | гильдии, метаданные | READY, GUILD_CREATE/UPDATE |
| `ChannelStore` | каналы всех гильдий и ЛС | READY, CHANNEL_CREATE/UPDATE |
| `MessageStore` | сообщения по каналам (окно) | MESSAGE_CREATE, история REST |
| `GuildMemberStore` | members по (guild,user) | GUILD_MEMBERS_CHUNK, MEMBER_ADD |
| `PresenceStore` | статусы и активности | PRESENCE_UPDATE |
| `VoiceStateStore` | кто в каких голосовых | VOICE_STATE_UPDATE |
| `ReadStateStore` | последнее прочитанное, счётчики | READY, MESSAGE_ACK |
| `PermissionStore` | вычисленные права | производный от ролей+overwrites |
| `SelectedChannelStore` | текущий открытый канал | навигация |
| `RelationshipStore` | друзья/блок | READY, RELATIONSHIP_ADD |
| `UserSettingsStore` | настройки пользователя | READY, USER_SETTINGS_UPDATE |

### Ожидаемое поведение памяти
Stores **не** держат всё вечно. `MessageStore` хранит **окно** сообщений вокруг
просматриваемой позиции (например, ±50–100), при скролле догружает и **выгружает**
дальние. `GuildMemberStore` для больших гильдий заполняется лениво (см. `REQUEST_GUILD_MEMBERS`).

## 1.4. Транспортный слой

### 1.4.1 REST-клиент
- Обёртка над `fetch`/XHR с базовым URL `/api/v10`.
- Автоматически добавляет `Authorization`, обрабатывает **rate limit** (читает
  `X-RateLimit-*`, ставит запросы в очередь по bucket'ам, ждёт `Retry-After` на 429).
- Ретраи с экспоненциальным backoff для сетевых ошибок (не для 4xx).

### 1.4.2 Gateway-клиент
- Держит один WebSocket. Управляет жизненным циклом: IDENTIFY/RESUME, heartbeat, реконнект
  с backoff, буфер последнего `seq`. (Полностью — глава 03.)

### 1.4.3 Voice-клиент
- Отдельное подключение при заходе в голос: WebSocket сигнализации + WebRTC/UDP медиа.
  (Полностью — глава 05.)

### 1.4.4 Локальный RPC-сервер
- Desktop поднимает **локальный WebSocket** на диапазоне портов `6463–6472`, к которому
  подключаются игры/приложения (Rich Presence, авторизация). (Полностью — глава 26.)

### Альтернативы транспорта
- **Один канал на всё** (например, только WebSocket) — Discord намеренно разделяет:
  REST для идемпотентных CRUD (кэшируемо, ретраибельно), Gateway для push-событий,
  UDP для медиа (задержка важнее надёжности). Смешивать их в один канал = худшие
  характеристики по каждому измерению.

## 1.5. Презентационный слой

### Виртуализация списков
- Список сообщений и список участников больших гильдий **виртуализируются**: в DOM
  находятся только видимые элементы (+ буфер), остальные заменяются распорками по высоте.
- **Ожидаемое поведение:** плавный скролл на десятках тысяч сообщений без роста DOM;
  сохранение позиции скролла при подгрузке истории сверху (anchor к элементу, а не к
  пиксельной позиции).
- **Технологии:** собственный virtualized-scroller (у сообщений переменная высота, поэтому
  нужен измеряющий/динамический виртуализатор, а не фиксированный).

### Роутинг
- Клиентский роутер: `/channels/@me/:dmId`, `/channels/:guildId/:channelId`,
  `/store`, `/discovery`. Настройки — оверлей поверх, не отдельный маршрут.

### Локализация (i18n)
- Строки вынесены в каталоги переводов, подстановка по ключам, поддержка плюрализации и
  интерполяции. Язык из настроек, fallback — английский.

## 1.6. Electron-специфика (desktop)

```
┌───────────────────────────────────────────────┐
│  MAIN PROCESS (Node.js)                        │
│  - создание окон (BrowserWindow)               │
│  - автообновление (host/module updater)        │
│  - оверлей поверх игр                           │
│  - глобальные хоткеи, tray, push-to-talk       │
│  - crash reporter, логи                         │
│         ▲ IPC ▼                                 │
│  RENDERER PROCESS (Chromium)                    │
│  - весь React/Flux/UI                           │
│  - изолирован (contextIsolation), доступ к ОС   │
│    только через безопасный preload-мост         │
└───────────────────────────────────────────────┘
```

### Что делает main-процесс
Единственный, у кого есть доступ к ОС: файлы, автозапуск, оверлей, глобальные хоткеи
(push-to-talk работает даже когда окно не в фокусе), управление окнами и обновлениями.

### IPC (главный ↔ renderer)
- Обмен через `ipcMain`/`ipcRenderer` + безопасный **preload** с `contextBridge`.
- **Ожидаемое поведение безопасности:** renderer НЕ имеет прямого `require`/Node-доступа
  (`nodeIntegration: false`, `contextIsolation: true`); всё общение — через явно
  объявленные каналы. Это защищает от XSS-эскалации (сообщение с вредоносным контентом
  не должно получить доступ к файловой системе).

### Технологии
- **Electron** (Chromium + Node). Discord мигрировал host-обвязку между версиями Electron,
  вынося тяжёлые нативные части (оверлей, голос) в отдельные нативные модули.
- Нативные модули на C++ (voice engine, overlay-инъекция).

### Почему Electron
- **Альтернатива Tauri/native (WebView2/WKWebView):** легче по памяти, но фрагментированный
  движок (разный WebView на разных ОС) → нестабильный рендеринг сложного UI. Discord выбрал
  Electron ради **единого предсказуемого Chromium** на всех платформах и переиспользования
  веб-кодовой базы 1:1.
- **Альтернатива полностью нативным клиентам** (Qt/C++, WinUI, AppKit) — максимальная
  производительность, но ×3 стоимость разработки и рассинхрон фич между платформами.
  Discord приоритезировал скорость выпуска фич и единый код.

> Примечание для rift-messenger: проект уже мигрировал с Tauri на Electron
> (см. историю коммитов) — этот выбор согласуется с архитектурой выше.

## 1.7. GPU-рендеринг

### Что делает
Перекладывает композитинг слоёв, скролл и анимации на GPU, разгружая CPU.

### Как работает / ожидаемое поведение
- Chromium композитит страницу слоями на GPU; свойства `transform`/`opacity` анимируются
  без layout/paint (только composite) → 60 fps.
- **Аппаратное ускорение видео**: декод входящих видеопотоков (VP8/9/H.264/AV1) и энкод
  исходящего (screen share/камера) идут через GPU-кодеки, а не CPU.
- **Ожидаемое поведение:** при включённом «Hardware Acceleration» падает загрузка CPU в
  звонках; при проблемных драйверах пользователь может отключить его в настройках →
  fallback на софтверный рендеринг.

### Альтернативы
- Софтверный рендеринг — совместимость выше, но CPU-нагрузка и энергопотребление хуже;
  поэтому GPU по умолчанию, с ручным откатом.

## 1.8. Оптимизация памяти на клиенте

Клиент такого масштаба обязан агрессивно управлять памятью.

### Приёмы и ожидаемое поведение
1. **Оконные хранилища** — сообщения/участники не держатся целиком (см. 1.3).
2. **Виртуализация DOM** — постоянный размер дерева независимо от объёма данных.
3. **Ленивая загрузка гильдий** — при READY гильдии приходят как «unavailable», их
   содержимое (участники, presence) подтягивается по мере открытия.
4. **Выгрузка неактивных** — presence/typing для несмотрящихся каналов не держатся детально.
5. **Разгрузка изображений** — вне вьюпорта картинки могут выгружаться из памяти декодера.
6. **`isFastConnection`/экономия** — на слабых сетях меньше префетча.

### Технологии
Профилирование через Chromium DevTools/heap snapshots; собственные метрики RSS в
main-процессе; принудительный GC в некоторых сценариях.

## 1.9. Локальное хранилище (клиент)

| Данные | Где хранятся | Зачем |
|--------|--------------|-------|
| Токен авторизации | защищённое хранилище ОС / зашифрованный стор | автологин |
| Настройки UI (тема, zoom) | локальный key-value | мгновенный старт до сети |
| Кэш эмодзи/аватаров | дисковый кэш Chromium + CDN-кэш | offline-превью, меньше сети |
| Черновики сообщений | локально по каналу | не терять недописанное |
| Логи | файлы в user-data | диагностика |
| Очередь аналитики | локально до отправки | не терять события офлайн |

### Ожидаемое поведение
- Токен **не** должен лежать в открытом `localStorage` в проде (риск кражи). Discord
  хранит его в защищённом сторе; веб-версия — в изолированном хранилище.
- Черновик восстанавливается при возврате в канал.

### Технологии
- Web: `IndexedDB`/`localStorage` (для веб-версии), Cache Storage.
- Desktop: user-data директория Electron, шифрование чувствительного через ОС-keychain.

### Альтернативы
- Хранение токена в cookie (HttpOnly) — для веба безопаснее от JS-кражи; Discord исторически
  использует локальный стор ради работы desktop RPC. Компромисс: строгий CSP + изоляция.

## 1.10. Автообновления клиента

### Что делает
Держит desktop-клиент актуальным без ручной переустановки.

### Как работает / ожидаемое поведение
- Двухуровневая схема: **host** (тяжёлая оболочка, обновляется редко, полный установщик)
  и **modules/core** (JS-бандл и нативные модули, обновляются часто, дельта-патчами).
- При старте и периодически клиент проверяет манифест версий на сервере обновлений; если
  есть новая — качает в фоне, применяет при следующем перезапуске (или предлагает
  перезапуститься баннером).
- **Ожидаемое поведение:** обновление незаметно, откат при сбое (если новый модуль не
  стартует — возврат к предыдущему), staged rollout (процент пользователей).

### Технологии
- Собственный updater поверх Electron autoUpdater/Squirrel-подобной схемы; дельта-обновления
  модулей; подпись пакетов.

### Альтернативы
- Полный установщик каждый раз — просто, но тяжело по трафику при частых релизах. Дельты
  на модулях дают быстрые ежедневные апдейты.

## 1.11. Логи, диагностика, crash reports (клиентская часть)

### Что делает
Собирает информацию для отладки проблем у пользователей.

### Как работает / ожидаемое поведение
- **Логи**: ротируемые файлы (voice, gateway, updater, renderer) в user-data. Уровни
  debug/info/warn/error. Пользователь может приложить их в тикет.
- **Crash reports**: при падении renderer/main формируется minidump + метаданные (версия,
  ОС, GPU) и отправляется в crash-бэкенд.
- **Diagnostics**: встроенный тест голоса/сети (проверка микрофона, echo test, качество
  соединения, WebRTC-статистика).
- **Ожидаемое поведение приватности:** содержимое сообщений в логи/краши не попадает;
  собираются технические метаданные, отправка краш-отчётов управляется настройкой.

### Технологии
- Crashpad/Breakpad (Chromium) для minidump; собственный симптоматический сборщик; Sentry-подобный
  бэкенд агрегации.

### Альтернативы
- Только серверные логи — недостаточно: большинство проблем (аудио-драйверы, GPU, сеть)
  видны только на клиенте. Поэтому клиентская телеметрия обязательна, но с фильтрацией PII.

## 1.12. Мобильная синхронизация (клиентская сторона)

### Что делает
Состояние (прочитанное, настройки, presence) консистентно между устройствами.

### Как работает / ожидаемое поведение
- **Read state синхронизируется через сервер**: прочтение канала на телефоне шлёт `ACK`,
  сервер рассылает `MESSAGE_ACK` на другие сессии пользователя → счётчики обнуляются везде.
- **Настройки** хранятся на сервере (`USER_SETTINGS_UPDATE`), а не только локально →
  одинаковый вид на всех клиентах.
- **Presence агрегируется по сессиям**: если пользователь онлайн на телефоне и desktop,
  сервер сводит статус; «mobile online» показывается отдельной иконкой.
- **Push-уведомления** на мобильном приходят, только если сессия не «активна» на другом
  устройстве (чтобы не дублировать сигнал, когда человек уже читает на десктопе).

### Технологии
- Серверный **read state service**, синхронизация настроек в PostgreSQL, push через
  APNs (iOS) / FCM (Android).

### Альтернативы
- Локальное хранение настроек без сервера — рассинхрон между устройствами. Discord держит
  источник истины на сервере, клиент — кэш.

## 1.13. Сводка: жизненный цикл запуска клиента

1. **Старт процесса** (Electron main) → создание окна, проверка обновлений.
2. **Renderer грузит бандл**, читает локальные настройки (мгновенный UI-скелет).
3. **REST login** по сохранённому токену → базовый профиль.
4. **Gateway connect** → IDENTIFY → **READY** (снимок: гильдии-заглушки, ЛС, настройки).
5. **Ленивое наполнение**: при открытии гильдии — участники/presence; при открытии канала —
   история сообщений (REST) + подписка на события.
6. **Голос** — только при заходе в голосовой канал.
7. **Фоновая синхронизация**: read state, presence, typing в реальном времени по Gateway.
