# 02 — Полная архитектура сервера

Серверная сторона Discord — это набор специализированных сервисов, каждый оптимизирован
под свой профиль нагрузки. Нет одного «монолита-бэкенда»; есть плоскости (REST, real-time,
media) и слой данных под ними.

---

## 2.1. Карта сервисов

```
                         ┌────────────────────┐
        клиенты  ─────►  │   Edge / LB / TLS   │
                         └─────────┬──────────┘
              ┌────────────────────┼─────────────────────┐
              ▼                    ▼                     ▼
      ┌──────────────┐     ┌──────────────┐      ┌──────────────┐
      │  REST API    │     │  Gateway     │      │ Voice/Media  │
      │  (HTTP)      │     │ (WebSocket)  │      │ (WebRTC/SFU) │
      └──────┬───────┘     └──────┬───────┘      └──────┬───────┘
             │                    │                     │
             ▼                    ▼                     ▼
      ┌───────────────────────────────────────────────────────┐
      │            СЛОЙ ДАННЫХ И ФОНОВЫХ СЕРВИСОВ               │
      │  ScyllaDB (сообщения) · PostgreSQL (пользователи,      │
      │  биллинг) · Redis (кэш/presence) · Elasticsearch       │
      │  (поиск) · Object Storage/CDN · очереди/воркеры        │
      │  · data services (Rust-кэш) · fanout · read state      │
      └───────────────────────────────────────────────────────┘
```

### Что делает
Каждая плоскость масштабируется независимо: REST — под пиковые CRUD-запросы, Gateway —
под миллионы постоянных соединений, Media — под голосовой трафик. Слой данных общий.

### Альтернативы и почему так
- **Монолит** — проще на старте, но нельзя масштабировать real-time отдельно от CRUD; при
  масштабе Discord один процесс не удержит миллионы WebSocket. Поэтому — разделение по
  профилю нагрузки.
- **Полный микросервисный зоопарк** — гибко, но операционно дорого. Discord держит
  относительно **крупные сервисы** по плоскостям (не «сервис на каждую сущность»), балансируя
  между масштабируемостью и простотой эксплуатации.

---

## 2.2. REST API-сервис

### Что делает
Обрабатывает все идемпотентные операции: аутентификация, отправка сообщений, управление
гильдиями/каналами/ролями, профили, загрузка вложений (выдача upload-URL).

### Как работает / ожидаемое поведение
1. Запрос проходит edge → аутентификация токена (проверка HMAC-подписи и срока).
2. **Rate limiting** по bucket'ам маршрута и глобально (см. 2.7).
3. Проверка **прав** (роли + overwrites) до любой мутации.
4. Валидация тела, выполнение операции, запись в БД.
5. Публикация соответствующего события в **fanout** (чтобы Gateway разослал real-time).
6. Ответ клиенту (часто — созданный объект целиком).

### Что происходит на сервере при отправке сообщения (пример сквозного пути)
```
POST /channels/{id}/messages
  → auth → rate-limit bucket(channel) → permission check (SEND_MESSAGES)
  → generate Snowflake → write ScyllaDB (partition = channel_id + time-bucket)
  → enqueue fanout(MESSAGE_CREATE, recipients = подписчики канала)
  → 200 + message object
Параллельно: fanout → Gateway-сессии получателей → WS push MESSAGE_CREATE
```

### Технологии
- Исторически **Python** (с оптимизациями), значимые части переписаны на **Rust** и **Go**
  ради латентности и памяти.
- HTTP-фреймворк + пул соединений к БД.

### Альтернативы
- **GraphQL** вместо REST — гибкие выборки, но кэширование и rate-limit сложнее, а нагрузка
  Discord «широкая и однотипная» (много одинаковых мелких запросов) → REST с чёткими
  bucket'ами предпочтительнее.

## 2.3. Gateway-сервис (real-time)

### Что делает
Держит постоянные WebSocket-соединения и рассылает события в реальном времени. Сердце
«живого» Discord. (Протокол детально — глава 03.)

### Как работает / ожидаемое поведение
- По процессу-актору на соединение; сессия хранит буфер последних событий для **RESUME**.
- Подписки: сессия получает только события гильдий/каналов, которые ей релевантны и на
  которые она подписана (intents + guild subscriptions).
- **Presence fanout**: при смене статуса пользователя событие нужно доставить всем, кто его
  «видит» (общие гильдии/друзья) — это самая тяжёлая рассылка.

### Технологии
- **Elixir/Erlang на BEAM** — модель лёгких процессов идеально ложится на миллионы
  конкурентных соединений; вытесняющая планировка не даёт одному соединению заблокировать
  других; распределённый Erlang связывает узлы.
- Для сверхгорячих рассылок Discord выносил часть логики (presence, guild subscriptions) в
  оптимизированные сервисы, в т.ч. на **Rust** (через Rustler-NIF), когда BEAM упирался в CPU.

### Альтернативы и почему BEAM
- **Node.js / Go** тоже держат много соединений, но BEAM даёт **изоляцию отказов**
  (падение одного процесса-соединения не роняет соседей), встроенную супервизию и
  распределённость — критично при миллионах сессий. Это исторический ключевой выбор Discord.

## 2.4. Слой данных

### 2.4.1 Сообщения — ScyllaDB (ранее Cassandra)
- **Что хранит:** триллионы сообщений.
- **Модель:** партиционирование по `(channel_id, bucket)`, где bucket — временнóе окно.
  Кластеризация по message Snowflake → сообщения физически лежат по времени, пагинация
  «до/после ID» дешёвая.
- **Ожидаемое поведение:** запись быстрая (LSM), чтение окна свежих сообщений — по одной
  партиции; удаление — tombstone; редактирование — новая версия.
- **Почему ScyllaDB:** совместима с Cassandra, но на C++ без пауз GC → предсказуемая
  латентность; решает проблему «хвостовых» задержек, что были на Cassandra/JVM.
- **Проблема hot partitions:** очень активный канал = горячая партиция. Решается слоем
  **data services** (Rust) — кэш и **coalescing** одинаковых запросов (много клиентов
  просят одно и то же — делается один запрос к БД, ответ раздаётся всем).

### 2.4.2 Реляционные данные — PostgreSQL
- Пользователи, регистрация, гильдии-метаданные, биллинг, подписки Nitro.
- Транзакции и целостность там, где они важнее масштаба записи.

### 2.4.3 Кэш и presence — Redis
- Сессии, presence-агрегация, временные счётчики, rate-limit-буфер.

### 2.4.4 Поиск — Elasticsearch
- Индексация сообщений для полнотекстового поиска (по гильдиям), с учётом прав доступа.

### 2.4.5 Object storage + CDN
- Вложения, аватары, эмодзи, стикеры. Клиент грузит напрямую в хранилище по выданному URL,
  в сообщении хранится ссылка. (Детально — глава 34.)

### Альтернативы слоя данных
- **Единая SQL-СУБД для всего** — не тянет объём сообщений (триллионы строк, гео-распределение).
  Разделение «SQL для сущностей + wide-column для сообщений» — стандартный масштабный паттерн.
- **MongoDB** (с чего Discord начинал) — не выдержал рост, отсюда переезд на Cassandra→ScyllaDB.

## 2.5. Fanout (рассылка событий)

### Что делает
Доставляет событие (новое сообщение, presence, typing) всем релевантным подключённым
сессиям.

### Как работает / ожидаемое поведение
- При событии определяется множество получателей (участники канала / кто видит presence).
- Событие маршрутизируется на те Gateway-узлы, где сидят их сессии, и пушится по WS.
- **Оптимизация для больших гильдий:** нельзя пушить presence каждого каждому (N² взрыв).
  Клиент подписывается на **видимый диапазон** участников (то, что реально на экране), и
  presence раздаётся выборочно (**lazy guild subscriptions**).

### Технологии
Внутренняя шина маршрутизации на BEAM + сервисы presence; для крупных рассылок —
оптимизированные пути.

### Альтернативы
- **Broadcast всем** — просто, но O(N²) на больших гильдиях кладёт сеть/CPU. Выборочные
  подписки — обязательное условие масштаба.

---

## 2.6. Sharding (на стороне сервера и ботов)

### Что делает
Делит нагрузку Gateway на части: каждый shard обслуживает подмножество гильдий.

### Как работает / ожидаемое поведение
- Гильдия закреплена за shard по формуле `shard_id = (guild_id >> 22) % num_shards`.
- Боты обязаны шардироваться при росте: открывают несколько WS-соединений, по одному на
  shard-диапазон.
- **Ожидаемое поведение:** событие гильдии приходит только на её shard; ЛС — на shard 0.

### Технологии / Альтернативы
- Альтернатива — один узел на всё: не масштабируется. Sharding по стабильному ключу
  (guild_id) даёт горизонтальное масштабирование без переезда гильдий между узлами.
  (Детально — глава 36.)

---

## 2.7. Rate limiting (серверная реализация)

### Что делает
Защищает от злоупотреблений и перегрузки, обеспечивает честность.

### Как работает / ожидаемое поведение
- **Per-route buckets:** каждому маршруту (+ major-параметру вроде channel_id) назначен
  bucket с лимитом на окно. Ответ несёт `X-RateLimit-Limit/Remaining/Reset/Bucket`.
- **Global limit:** суммарный потолок запросов на токен (~50/с).
- **Gateway limits:** ограничение IDENTIFY (сессий/мин) и исходящих команд (~120/60с).
- При превышении — **429** с `Retry-After`; повтор нарушений → временный бан/CloudFlare.

### Технологии
Счётчики в Redis (атомарные инкременты с TTL), sliding/leaky-bucket.

### Альтернативы
- Фиксированное окно — проще, но даёт всплески на границе окна. Leaky/token-bucket ровнее.

## 2.8. Read state service

### Что делает
Отслеживает, что пользователь прочитал, и синхронизирует между устройствами.

### Как работает / ожидаемое поведение
- Хранит по (user, channel) последний прочитанный message ID и счётчик упоминаний.
- Клиент шлёт ACK при прочтении → сервис обновляет → рассылает `MESSAGE_ACK` на другие
  сессии пользователя (мультидевайс-синхрон, см. 01.12).
- Вычисляет «непрочитанные» и бейджи упоминаний.

### Технологии
Оптимизированное key-value хранилище + real-time рассылка.

---

## 2.9. Фоновые воркеры и очереди

### Что делают
Асинхронная обработка: доставка push-уведомлений, генерация превью/тумбнейлов, обработка
вложений, AutoMod-сканирование, аналитика, отправка email, начисления Nitro/boost.

### Ожидаемое поведение
Тяжёлые/медленные задачи не блокируют горячий путь запроса: REST-хендлер публикует задачу
в очередь и отвечает, воркеры обрабатывают асинхронно. Гарантии at-least-once + идемпотентность
воркеров.

### Технологии
Очереди сообщений (Kafka/подобное), пулы воркеров.

---

## 2.10. Медиа-плоскость (обзор; детали — гл. 05)

### Что делает
Голос, видео, демонстрация экрана. Отдельная инфраструктура голосовых серверов (SFU).

### Ключевые свойства
- Гео-распределённые голосовые регионы; клиент подключается к ближайшему/оптимальному.
- **SFU** (Selective Forwarding Unit): принимает поток каждого участника и **пересылает**
  остальным (не микширует централизованно, не mesh).
- Транспорт — **UDP/WebRTC**, аудио — **Opus**, шифрование — libsodium.

### Технологии
C++/Rust медиасерверы поверх libwebrtc.

---

## 2.11. Надёжность, деплой, наблюдаемость

### Ожидаемое поведение
- **Репликация БД** и мультизональность; отказ узла не теряет данные.
- **Graceful reconnect:** при рестарте Gateway-узла сессии переключаются на другой узел и
  делают RESUME без полной перезагрузки клиента.
- **Staged rollout** релизов, feature flags, канареечные развёртывания.
- **Мониторинг:** метрики латентности/ошибок по сервисам, трейсинг сквозных запросов,
  алертинг.

### Технологии
Оркестрация контейнеров, сервис-меш/собственный роутинг, Prometheus-подобные метрики,
распределённый трейсинг.

## 2.12. Сквозная сводка: путь одного действия

**Пользователь A пишет в канал, пользователь B (онлайн) должен мгновенно увидеть:**

1. A → `POST /channels/{id}/messages` (REST).
2. REST: auth → rate-limit → permission (SEND_MESSAGES) → Snowflake → запись в ScyllaDB.
3. REST → fanout(`MESSAGE_CREATE`, получатели канала).
4. Fanout находит Gateway-сессию B → push по WebSocket.
5. Клиент B: Gateway-событие → Flux action → MessageStore/ReadState → перерисовка.
6. Если B офлайн на десктопе, но онлайн на телефоне и не активен → воркер шлёт push (APNs/FCM).
7. A получает HTTP-ответ, сопоставляет по `nonce`, снимает статус «sending».
