# Discord Clone — Полная техническая спецификация

> Техническое задание уровня продукта масштаба Discord. Документ предназначен для команды
> разработки, по которой можно строить продукт практически с нуля.
>
> **Формат каждого раздела (шесть линз):**
> 1. **Что делает** — назначение функции.
> 2. **Как работает / ожидаемое поведение** — механика и UX.
> 3. **Что происходит на клиенте** — UI, состояние, локальная логика.
> 4. **Что происходит на сервере** — API, БД, обработка.
> 5. **Технологии** — конкретный стек.
> 6. **Альтернативы и почему выбран этот подход** — инженерные компромиссы.

Целевой проект: **rift-messenger**. Спецификация пишется как отдельный, самодостаточный
документ, но с прицелом на реализацию в этой кодовой базе.

## Как читать документ

- Разделы можно читать независимо, но 1–3 (клиент, сервер, gateway) — фундамент,
  на который ссылаются все остальные.
- Термины даются с англоязычным оригиналом в скобках, чтобы совпадать с реальным API Discord.
- Числовые значения (лимиты, размеры, битрейты) — ориентиры на основе публичного поведения
  Discord; для клона их можно менять, но они задают порядок величин.

## Глоссарий базовых терминов

| Термин | Значение |
|--------|----------|
| **Guild** | «Сервер» в терминах API — сообщество с каналами, ролями, участниками. |
| **Channel** | Канал внутри гильдии или ЛС (текст, голос, форум, stage…). |
| **Snowflake** | 64-битный уникальный ID со встроенным timestamp. |
| **Gateway** | WebSocket-соединение реального времени клиент↔сервер. |
| **Dispatch** | Событие, которое Gateway рассылает клиентам (`MESSAGE_CREATE` и т.д.). |
| **Intent** | Битовая подписка на категорию событий Gateway. |
| **Shard** | Один Gateway-процесс, обслуживающий подмножество гильдий. |
| **SFU** | Selective Forwarding Unit — медиасервер, пересылающий потоки участников. |
| **Presence** | Онлайн-статус + активность пользователя. |
| **Embed** | Богатая карточка внутри сообщения. |
| **Member** | Связка пользователь↔гильдия (ник, роли, дата входа). |
